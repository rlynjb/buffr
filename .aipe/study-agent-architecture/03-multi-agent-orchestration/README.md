# SECTION C — Multi-agent orchestration

Anchor: multi-agent (primary). The load-bearing new ground — taught as
new material.

**buffr is single-agent, so this entire section is study material,
honestly marked "Not yet implemented."** It is generated in full because
the boundary file is load-bearing and the topologies are the genuinely
new ground above one agent. Where buffr's future (the two-brain
laptop+phone design in `agent-layer-plan.md`) would adopt a topology,
each file names which one and why.

## Reading order

1. `01-when-not-to-go-multi-agent.md` — **read first.** The most
   important multi-agent decision is whether to be multi-agent. buffr
   correctly is not.

   adopt (laptop boss + phone worker).
3. `03-sequential-pipeline.md` — workers in series.
4. `04-parallel-fan-out.md` — workers in parallel (buffr has the
   `Promise.all` primitive at the I/O layer).
5. `05-debate-verifier-critic.md` — the bundle's unused `rubric-judge`.
6. `06-swarm-handoff.md` — recognized, a non-fit for buffr's
   trajectory-first design.

   human-in-the-loop.
8. `08-shared-state-and-message-passing.md` — buffr already has a
   data-layer blackboard (`chunks`).
9. `09-coordination-failure-modes.md` — the "2-5x tax" made concrete;
   buffr's single-agent caps are the seeds of the bounds.
