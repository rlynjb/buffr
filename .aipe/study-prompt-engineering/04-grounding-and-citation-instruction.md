# Grounding and citation instruction

*Grounded generation / cite-your-sources prompting — Industry standard (the RAG
"answer only from retrieved context, cite it" instruction).*

## Zoom out, then zoom in

Here's the thing nobody tells you about "cite your sources" prompts: the
instruction is the *weak* half. The reason grounding works in this repo isn't that
`BASE_SYSTEM` sternly says "cite sources" — it's that the search tool hands the
model **pre-formatted citations it just copies.** The prompt asks; the tool
provides. Understand that and you understand why this is reliable *and* why it's
not actually enforced.

```
  Zoom out — where grounding sits

  ┌─ aptkit agent ───────────────────────────────────────────┐
  │  ★ BASE_SYSTEM: "call search first, ground, cite" ★       │ ← the instruction
  └───────────────────────────┬──────────────────────────────┘
                             │ drives a tool call
  ┌─ retrieval (this repo's data) ▼──────────────────────────┐
  │  search_knowledge_base → returns "[docId] snippet"        │ ← the citations
  └───────────────────────────┬──────────────────────────────┘
                             │ tool_result back into the prompt
  ┌─ model ─────────────────────▼────────────────────────────┐
  │  copies the [docId] tags into its prose answer            │
  └───────────────────────────────────────────────────────────┘
```

Zoom in: the concept is **grounded generation with citation** — instruct the model
to answer only from retrieved context and to cite it, and make that easy by handing
back citation-shaped results. The question it answers: *how do you get an answer
tied to sources instead of hallucinated?*

## Structure pass

Trace one axis — **who provides the citation text?** — and find the surprise: it's
not the model.

```
  Axis: "where does the [docId] citation come from?"

  ┌─ BASE_SYSTEM ─────┐  provides: the INSTRUCTION to cite ("cite their sources")
  └─────────┬─────────┘
          ══╪══ seam: instruction → data  (the citation TEXT is not authored here)
  ┌─ search tool ─────▼┐ provides: the citation STRING — "[docId] snippet"
  │ toResult()        │  pre-built, the model only has to copy it
  └─────────┬─────────┘
          ══╪══ seam: data → output  (NOTHING checks the model actually copied it)
  ┌─ final answer ────▼┐ provides: prose that MAY OR MAY NOT include the [docId]
  └───────────────────┘   citation is UNENFORCED
```

Two seams, two lessons. First seam: the instruction and the citation text have
different authors — `BASE_SYSTEM` says "cite," the tool writes the actual `[docId]`.
Second seam, the one to be honest about: **nothing validates that the model's final
answer contains the citation.** It's an instruction the model usually follows
because copying is easy, not a constraint the system enforces.

## How it works

### Move 1 — the shape

Think of a fill-in-the-blank form where the blanks come pre-labeled. The tool
returns `[work.md] The author is a software engineer…` — the model's easiest path
to a fluent answer is to keep the `[work.md]` tag attached to the fact. Grounding
works by making the cited form the *lowest-effort* form.

```
  Pattern — citation handed to the model, not generated by it

  BASE_SYSTEM:  "ground every answer in retrieved chunks, cite their sources"
                          │ triggers
                          ▼
  search tool returns:  results: [ { citation: "[work.md] The author is a…" }, … ]
                          │ tool_result message
                          ▼
  model writes:  "You're a software engineer [work.md]."   ← copied the tag
```

### Move 2 — the walkthrough

**The instruction (aptkit's `BASE_SYSTEM`).** Four sentences, doing four jobs.

```ts
// rag-query-agent.js:12-19
const DEFAULT_SYSTEM_TEMPLATE = [
  'You are a personal knowledge assistant.', '',
  `Always call the ${SEARCH_KNOWLEDGE_BASE_TOOL_NAME} tool first to retrieve relevant`,  // tool-first
  'passages before answering. Ground every answer in the retrieved chunks and cite',     // ground + cite
  'their sources. If the knowledge base does not contain the answer, say so plainly',     // refuse-to-guess
  'rather than guessing.',
].join('\n');
```

One job per sentence — the spec's decomposition rule, well applied: (1) identity,
(2) tool-first ordering, (3) ground-and-cite, (4) refuse to hallucinate. The
refuse-to-guess clause is the under-appreciated one — it's what turns "I don't know"
into an acceptable output instead of a hallucinated one.

**The citation the tool hands back (retrieval).** This is the load-bearing half.
Every search result is pre-formatted as `[docId] snippet`.

```ts
// search-knowledge-base-tool.js — toResult()
function toResult(hit) {
  const docId = typeof hit.meta.docId === 'string' ? hit.meta.docId : hit.id;
  const text = typeof hit.meta.text === 'string' ? hit.meta.text : '';
  const snippet = text.length > 160 ? `${text.slice(0, 157)}...` : text;
  return {
    id: hit.id, score: hit.score,
    citation: snippet ? `[${docId}] ${snippet}` : `[${docId}]`,   // ← pre-built citation
    meta: hit.meta,
  };
}
```

The `[${docId}]` wrapper is the trick. The model doesn't *invent* a citation format
and doesn't have to *track* which source a fact came from — the source label is
welded to the snippet text before the model ever sees it. Copying the welded tag is
easier than stripping it, so the model tends to keep it. That's grounding-by-
ergonomics, not grounding-by-instruction.

**The result flows back as a tool message.** The agent loop pushes the JSON-encoded
results back as a `tool_result`, which becomes the context the synthesis turn
answers from.

```ts
// run-agent-loop.js:79, 97-104
resultContent = truncate(JSON.stringify(result));     // the [docId] citations, serialized
toolResults.push({ type: 'tool_result', toolUseId: toolUse.id, content: resultContent });
messages.push({ role: 'user', content: toolResults }); // context for the next turn
```

Note: retrieved **conversation memory** arrives through this *exact same path* — it's
the same `search_knowledge_base` tool, just returning `kind:'memory'` chunks
(`conversation-memory.js`, `audit.md §12`). So "what we discussed earlier" gets
cited the same way "what's in your notes" does.

```
  Layers-and-hops — citation round trip

  ┌─ BASE_SYSTEM ─┐ hop 1: "cite sources"   ┌─ search tool ───┐
  │ instruction   │ ──drives tool call────► │ toResult():     │
  └───────────────┘                         │ "[docId] snip"  │
                                            └────────┬────────┘
                       hop 2: tool_result (JSON of citations) ▼
                                            ┌─ agent loop ────┐
                       hop 3: pushed as context message       │
                                            └────────┬────────┘
                       hop 4: model copies [docId] into prose ▼
                                              final answer (unverified)
```

### Move 3 — the principle

**Grounding is won at the tool boundary, not in the system prompt.** The
instruction sets intent; the citation-shaped tool result makes compliance the path
of least resistance. The honest corollary the spec demands you state: because
nothing validates the answer against the sources, this is *reliable-by-ergonomics,
not enforced.* If you needed a hard guarantee, you'd post-validate that every fact
in the answer maps to a returned `docId` — buffr doesn't, and for a personal KB
that's an acceptable call.

## Elaborate

This is textbook RAG grounding — the pattern AdvntrCue exercises too. The reference
material (Anthropic's "grounding in retrieved context," the OpenAI RAG cookbook)
all converges on the same move: retrieve, stuff the context, instruct the model to
answer only from it and cite. The buffr-specific refinement worth keeping is the
*pre-formatted citation string* — many implementations return raw chunks and ask
the model to construct citations, which is exactly where models drift and invent
sources. Welding `[docId]` to the snippet in `toResult` removes that degree of
freedom. The unenforced-citation gap is shared with most production RAG; the rigor
ceiling (validate-citations-against-sources) is where `study-ai-engineering`'s eval
discussion and `study-security`'s trust-boundary audit pick it up.

## Interview defense

**Q: "How does the assistant stay grounded instead of hallucinating?"**
Two halves, and the surprising one carries the weight. The system prompt says "call
search first, answer only from retrieved chunks, cite them, and if it's not there
say so." But the *reason* it works is the tool: it returns citations pre-formatted
as `[docId] snippet`, so the model copies a label instead of inventing one. Be
honest about the ceiling — nothing validates the final answer actually contains the
citation. It's grounding by ergonomics, not enforcement.

```
  instruction (cite) + tool returns [docId] (copyable) → grounded answer
                                                          (unverified)
```

model, not generated by it."*
